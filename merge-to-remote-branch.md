# Merge to Remote Branch

両方ともリモートブランチで、dev の方が 116-readme よりもコミットが進んでいる状況ですね。
そして、そのコミットの遅れを解消したい、つまりリモートの 116-readme をリモートの dev の状態に合わせたいということですね。

この状況で最も直接的かつ一般的な方法は、リモートの dev の内容をリモートの 116-readme にプッシュすることです。

## 基本的な解決策

ローカルに dev ブランチが最新の状態として存在しているという前提で進めます。

### ローカルの dev ブランチを最新にする

まず、ローカルの dev ブランチがリモートの dev の最新の状態であることを確認します。

```bash
git checkout dev # devブランチに切り替える
git pull origin dev # リモートのdevから最新の変更をプルする
```

### ローカルの dev をリモートの 116-readme にプッシュする

これでローカルの dev が最新の状態になったので、この内容をリモートの 116-readme にプッシュします。

```bash
git push origin dev:116-readme
```

このコマンドは、ローカルの dev ブランチが指すコミットを、リモートの 116-readme ブランチが指すように更新します。

## プッシュが拒否された場合（Fast-forward ではない場合）

もし、リモートの 116-readme ブランチに、あなたのローカルの dev にはないコミット（例えば、他の人が直接 116-readme にコミットした変更など）が存在する場合、上記の git push コマンドは拒否される可能性があります。
これは、履歴が分岐しているため、Git が「fast-forward」での更新ができないと判断するからです。

このような場合、以下の 2 つの選択肢があります。

### 1. マージしてからプッシュする（推奨、履歴を保持）

これは最も安全な方法で、リモートの 116-readme の既存の履歴を保持しつつ、dev の変更を統合します。

116-readme の内容をローカルにプルする

まず、リモートの 116-readme の内容をローカルに取り込みます。

```bash
git checkout 116-readme # ローカルの116-readmeブランチに切り替える
git pull origin 116-readme # リモートの116-readmeから変更をプルする
```

#### dev ブランチの内容を 116-readme にマージする

dev ブランチの変更を 116-readme に統合します。

```bash
git merge dev # devブランチの内容を現在の116-readmeブランチにマージする
```

ここで競合 (conflict) が発生する可能性があります。その場合は、競合を解決し、再度コミットしてください。

#### 116-readme をリモートにプッシュする

マージと競合解決が完了したら、ローカルの 116-readme をリモートにプッシュします。

```bash
git push origin 116-readme # ローカルの116-readmeをリモートにプッシュする
```

これで、リモートの 116-readme は dev の変更を取り込んだ状態になります。

### 2. 強制プッシュ（注意！履歴を上書き）

リモートの 116-readme ブランチの履歴を、ローカルの dev ブランチの履歴で完全に上書きしたい場合にのみ使用します。リモートの 116-readme に存在する、ローカルの dev にはないコミットは失われます。

この操作は、共有ブランチや他の開発者が作業しているブランチに対しては、絶対に避けるべきです。 状況によっては、他の開発者の作業を台無しにしてしまう可能性があります。

```bash
git push origin dev:116-readme --force
# または
git push origin dev:116-readme -f
```

どちらの方法を選ぶかは、116-readme ブランチに他の人の作業が含まれているか、またその履歴を保持する必要があるかどうかに依存します。
基本的には、1. マージしてからプッシュする 方法が安全でおすすめです。
